# 上海文旅景点 GEO 工作流 · 开工 Brief

> 本文件是项目权威 brief（SSOT）。仓库治理摘要见根目录 `CLAUDE.md`。
> 引擎复用自已验证的 LilyGEOMaster（礼盒 GEO），本项目把"实体"从【品牌】泛化为【景点】。

## 0. 一句话目标

搭一套可审计、可复现的智能 GEO 工作流，自动跑通「机会发现 → 选点/选主题 → 策略评估 → 内容运营 → 监测维护」的闭环，回答一个核心问题：**当游客问豆包"上海去哪玩"，AI 推荐了谁、谁没被推荐、哪些问题是没人占的内容空位。** 落地对象：上海文旅景点 / 目的地。

## 1. 必须先内化的领域模型

- **旅游 GEO 是什么**：让 AI 答案引擎（豆包 / ChatGPT / Perplexity）在回答游客问题时，主动**点名推荐某个景点/目的地/路线**，而不是在传统搜索/OTA 列表里排名。核心指标从"OTA 排名/搜索排名"变成 **景点占答率 / 提及率 / 首选推荐率 / 内容空位**。
- **谁会用 / 价值在哪**：
  - **景区/目的地方**：想知道自己在 AI 答案里的可见度、谁是 AI 眼里的"上海必去"、怎么进 AI 的推荐位。
  - **文旅服务商/创业**：把"测量 AI 可见度 + 找内容空位 + 产权威内容"做成可交付服务（对应 memory 里的 GEO 服务创业方向）。
- **三类游客客群（系统第一个要回答的，不预设）**：
  - **A = 外地国内游客**（来上海玩，问"必去/攻略/行程"）→ 中文 AI（豆包）。**主战场，量最大。**
  - **B = 入境外籍游客**（老外来上海）→ 英文 AI（ChatGPT/Perplexity，待 key）。
  - **C = 本地客**（上海人周末/约会/遛娃/小众）→ 中文 AI（豆包），需求与外地客完全不同。
- **AI 答案的"货架"在哪（决定打法）**：游客问 AI 之前，AI 的答案来自它读过的内容。中文侧主要内容源 = 小红书 / 携程马蜂窝 / 大众点评 / 抖音 / 各类攻略站。**你的景点/内容铺在这些平台上、被 AI 读到，才可能进答案。**
- **诚实约束（最重要）**：本系统只做**测量与辅助**，不能也不会"保证进答案"或"刷好评/铺假攻略"。AI 答案我们无法直接写入；能做的是把「测量 → 找空位 → 产真权威内容（真实游玩信息）→ 发布 → 再测量」这个闭环自动化。刷量、铺假软文 = 脆弱 + 会被模型降权 + 违广告法/平台规则，禁止。

## 2. 核心原则与红线（写进每个 agent 的系统约束）

1. **证据优先，拒绝幻觉**：所有指标从存档的原始证据（豆包回答原文 + 时间戳 + 原始响应体）计算，可回溯到证据 ID。不凭模型印象编数字。
2. **可解释、可复现**：同输入跑两次结论一致；每个评分附"为什么 + 证据 ID"。景点抽取**不**用 LLM（确定性查 watchlist 白名单）。
3. **HITL 闸门**：① 花钱前 ② 对外发布前 必须人审。默认**不**自动发布。
4. **合规 / ToS**：优先官方 API（火山方舟-豆包）；浏览器抓取封装在限速、守 ToS 的 adapter，绝不绕登录/付费墙。
5. **kill-criteria**：每个客群/主题设里程碑 + 止损线，跑不动就砍。
6. **景点真实信息发布前核实**：门票/时间/预约/交通等事实项会变，发布前必须核实，不编造。

## 3. 架构

- **真相面 (truth plane) = git**：代码 / 配置 / 证据库（原始抓取）/ 内容草稿，全部版本化、可审计。
- **控制面 (control plane) = Notion / 报表**：机会图、景点占答排行、内容看板、审批闸门。
- **Engine Adapter**：每引擎一个 `query() → capture() → parse()` connector，内置限速。豆包 adapter 复用自 Lily。
- **Evidence Store**：每条豆包回答存档，所有指标是对它的**纯函数**（`geo/metrics`）。
- **实体 = 景点**：`named_brands` 字段（引擎通用）在本项目承载景点实体；白名单 = `config/watchlist.yaml`。

## 4. 四大模块功能规格

### 4.1 机会发现 (Reconnaissance)
- **输入**：上海旅游高意图问题集（按客群×主题×区域×时长×人群×季节 fan-out，当前 ~94 条）+ 客群假设（A/B/C）。
- **做**：通过豆包 adapter 逐条提问 → 抓回答存证据 → 确定性解析"推荐了哪些景点、谁排第一、有没有空位"。
- **输出 → 机会图**：query × 竞争强度 × 客群 × 内容空位评分（AI 答得泛/无景点=高分机会）+ 景点占答时间线。

### 4.2 选点 / 选主题 (Selection)
- **做**：给候选（景点 or 内容主题）打"GEO 可赢度"= 内容空位 + incumbent 权威缺口 + 客群价值。
- **输出 → 排序短名单** + go/no-go + 理由（附证据 ID）。

### 4.3 策略评估 (Strategy)
- **做**：逐项评 该铺哪个平台（小红书/携程/抖音）、该产什么内容（攻略/路线/避坑/对比）、里程碑 + 止损线。
- **输出 → 每个主题一份策略备忘**。

### 4.4 内容运营 + 监测 (Ops & Maintenance)
- **内容流水线**：针对空位 query 产 AI 爱抓的内容（要点先行 / 结构化 / 真实信息 / 路线清晰）。人审后发布。
- **监测**：定时重跑 recon，追景点占答率与新鲜度；掉了告警；新对手进答案就标记。
- **闭环**：测量 → 找空位 → 产内容 → 发布 → 再测量 → 迭代。

## 5. 证据数据模型（复用 Lily schema，实体语义=景点）

每条 capture 至少含：`id, engine, query, buyer_segment(A/B/C), timestamp, raw_answer, named_brands[](=景点), cited_sources[], links[], is_mock, engine_model`。
所有指标（景点占答率/提及率/首选推荐率/空位评分）= 对该表的纯函数，禁止旁路。

## 6. 分阶段交付

- **Phase 0 · 引擎迁移 + 上海化**：复用引擎、定景点 watchlist、定客群、最小切片真跑。DoD：一条 query 抓回答、抽出景点、存证据。
- **Phase 1 · Recon 引擎**：上海问题集批量真侦察（A 主），出景点占答排行 + 机会图。
- **Phase 2 · 选点/选主题 + 策略**：可赢度短名单 + 策略备忘。
- **Phase 3 · 内容流水线**：针对空位产结构化攻略草稿，走人审。
- **Phase 4 · 监测 + 调度**：周期重跑、占答时间序列、告警。

## 7. 技术约束

- Python + pydantic v2 + httpx + tenacity + pytest（复用 Lily 栈）。
- API-first；secrets 走 env，绝不进仓库。
- adapter 限速 + 重试（429 退避）+ 单条容错；抓取守 ToS。
- 每个模块可单测；证据计算函数必有测试。
- **豆包接口选择**：用普通 `/chat/completions`（基础模型即给景点推荐，是旅游 GEO 核心信号）；联网 `/bots` 接口有日配额墙(429)，仅在需要引用源情报时另跑。
