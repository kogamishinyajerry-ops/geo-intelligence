# GEO 智能工作流 · 开工 Brief

> 本文件是项目权威 brief（SSOT）。仓库治理摘要见根目录 `CLAUDE.md`。

## 0. 一句话目标

搭一套可审计、可复现的智能 GEO 工作流，自动跑通「机会发现 → 选品 → 策略评估 → 内容运营 → 监测维护」的闭环，让任何同事都能在一个简单的控制台上看懂机会、一键批内容、看懂 KPI。第一个落地品类：面向商务场景的高端伴手礼盒。

## 1. 必须先内化的领域模型

- **GEO 是什么**：让 AI 答案引擎在回答里主动引用 / 推荐我们，而不是在传统搜索页排名。核心指标从"排名"变成 **占答率 / 提及率 / 首选推荐率**。引用权威会随时间复利——早占位，越往后越难被挤掉。
- **关键基础设施事实（决定一切打法）**：
  - 国内：豆包的商品卡只导向抖音商城 / 本地生活。进豆包答案的真实路径 ＝ 在抖音商城有店（评分 4.8+）+ 抖音生态种草 + 豆包 GEO 引擎做"需求—商品"匹配。淘宝接了千问，夸克未来大概率导向淘宝天猫。这是几个围墙花园——你站哪个货架，就决定你能进哪个 AI。
  - 跨境：英文 AI（ChatGPT / Perplexity / Copilot / Google AI）主要从 Shopify / 独立站 + 商品 feed 取货。Shopify Agentic Storefronts 一次设置可同步到三家。排序信号 ＝ 结构化商品数据、真实评价、准确价格库存、被引用的权威内容。
- **买家分叉（系统要回答的第一个问题，别预设）**：
  - 买家是「中国人买来送老外客户」→ 中文 AI + 抖音 / 淘宝货架；
  - 买家是「老外自己买」→ 英文 AI + Shopify。
  - 两者战场完全不同，英文那条大概率更空、客单更高、毛利更厚。让 recon 数据来定这一步，而不是拍脑袋。
- **打法是两手而非二选一（barbell）**：先"蹭"（带货返佣 / 给对手供货 / 做上游对比内容）试水 + 赚现金流并摸清空位；同时在"无人区"自建（对应货架 + 自有引用权威）。重心随权威累积，从蹭逐步移到自建。
- **诚实约束（最重要）**：本系统只做测量与辅助，不能也不会"保证进答案"或"注入广告"。AI 答案我们无法直接写入；能做的是把「测量 → 找空位 → 产出真权威内容 → 发布 → 再测量」这个闭环自动化。花钱买植入、批量铺假软文 ＝ 脆弱 + 会被模型降权 + 碰广告法，禁止。

## 2. 核心原则与红线（写进每个 agent 的系统约束）

- **证据优先，拒绝幻觉**：所有指标必须从存档的原始证据（抓取到的 AI 回答原文 + 时间戳 + 截图 + 抽取出的实体 / 引用 / 链接）计算得出。任何评分都要能回溯到"哪条证据支持"。不允许凭模型印象编数字。
- **可解释、可复现**：同样输入跑两次，结论一致；每个机会分 / 选品分 / 策略建议都附"为什么 + 依据证据 ID"。
- **人在环 (HITL) 闸门**：① 花钱前 ② 对外发布前，必须人工审批。系统默认不自动发布。
- **合规与 ToS**：优先用官方 API（OpenAI / Perplexity / 火山引擎-豆包 / Google；以及现成 AI 可见度监测 SaaS 的 API，如 Profound、Otterly、Ahrefs Brand Radar 等，按需选型）。任何浏览器抓取都封装在限速、守 ToS 的 adapter 里，绝不绕过登录 / 付费墙。广告内容依法标注。
- **纪律 / kill-criteria**：每个品类 / 动作设里程碑和止损线，跑不动就砍，自动化不许在死项目上空转。
- **简单到同事能用**：控制台要让非技术同事也能看懂机会图、一键批内容、看懂 KPI。复杂留在后端，前台只留清楚的卡片和按钮。

## 3. 架构

- **真相面 (truth plane) = GitHub**：所有代码、配置、证据库（原始抓取 + 截图）、内容草稿、产出物，全部版本化、可审计。
- **控制面 (control plane) = Notion（已接 MCP）**：机会图、选品清单、策略备忘、内容看板（待审 / 已发 / 待刷新）、KPI 仪表盘、审批闸门。同事在这里操作。
- **Engine Adapter 接口**：每个 AI 引擎一个可插拔 connector，统一实现 `query()` → `capture()` → `parse()`，内置限速。新增引擎 ＝ 加一个 adapter，不动主流程。
- **Agent / Skill 团队**：`recon`、`analysis`、`selection`、`strategy`、`content`、`publishing/feed`、`monitoring`、`reporting`，外加一个按计划跑闭环的 `orchestrator`。
- **Evidence Store**：每条 AI 回答都存档，所有指标由此派生（schema 见 §5）。

## 4. 四大模块功能规格

### 4.1 机会发现 (Reconnaissance)

- **输入**：品类种子词 + 目标买家假设（A / B 都跑）。
- **做**：扩展买家高意图问题（fan-out）→ 通过 adapter 在目标引擎逐条提问 → 抓回答存证据 → 解析"谁被点名、引用了哪些源、有没有链接 / 商品卡、占答率、情感"。
- **输出 → Notion 机会图**：query × 竞争强度 × 意图 × 该走的货架 × 空位评分（AI 答得泛 / 无品牌 ＝ 高分机会），并带竞争对手的占答率时间线。

### 4.2 选品 (Product Selection)

- **输入**：机会图 + 供给 / 成本 / 履约 / 我方设计能力。
- **做**：给候选细分打分 ＝ 需求（意图 × 覆盖）+ 竞争缺口（占答率空位）+ 适配（我方能力 / 供应链）+ 毛利 + GEO 可赢度（是否 AI 爱推的"需对比研究"型品类）。
- **输出 → 排序短名单** + 每项 go / no-go + 理由（附证据 ID）。

### 4.3 策略评估 (Strategy)

- **输入**：短名单。
- **做**：逐项评 自建 vs 蹭（成本、首次收入时间、天花板、可防御性、风险）；定货架与引擎（买家分叉的结论）；给分阶段计划 + 里程碑 + 止损线。
- **输出 → 每个细分一份策略备忘** + 组合层面的资源分配建议。

### 4.4 自动化运营 + 维护 (Ops & Maintenance)

- **内容流水线**：按目标 query 产出 AI 爱抓的内容（要点先行 / 结构化 / 真实数据 / 跨平台实体一致）——中文走知乎 / 小红书 / 公众号 / 百科 / 头条 + 抖音脚本；英文走 buying guide / product schema / 评价。人工审批后才进发布。
- **发布与 feed**：推送已批内容；管理商品 feed（Shopify Catalog / Google Shopping / ChatGPT feed / 经抖音进豆包）。
- **监测维护**：定时重跑 recon，追占答率与引用新鲜度；排名掉了告警；内容过 ~3 个月提醒刷新；发现新对手进答案就标记。
- **报表**：KPI 滚动到 Notion——占答率 / 提及率 / 首选推荐率、情感、AI 带来的流量与转化、每个细分的 P&L。
- **闭环**：测量 → 找空位 → 产内容 → 发布 → 再测量 → 迭代。自动化测量 + 调度 + 报表，内容创作保持人审。

## 5. 证据数据模型（可审计的地基，先把这个定死）

每条 capture 记录至少含：

```
id, engine, query, buyer_segment(A/B), timestamp,
raw_answer, cited_sources[], named_brands[], product_cards[],
links[], sentiment, screenshot_path
```

所有指标（占答率、提及率、首选推荐率、share-of-answer 时间序列）都是对该表的纯函数计算，禁止旁路。

## 6. 分阶段交付（小步快跑，每阶段先验证证据是真的再往上盖）

- **Phase 0 · 脚手架**：仓库、配置、Evidence Store schema、1 个 engine adapter、Notion 控制板骨架。DoD：能对 1 条 query 抓回答、存证据、在 Notion 看到。
- **Phase 1 · Recon 引擎**：query 生成 + 2–3 个 adapter（建议先 Perplexity API + 豆包）+ 解析 + 机会图。DoD：跑通礼盒品类的中 / 英双侦察，机会图里能看到空位评分和对手占答率。
- **Phase 2 · 分析 / 选品 / 策略**：打分 + 备忘生成。DoD：产出一份可读的选品短名单 + 策略备忘，每条带证据 ID。
- **Phase 3 · 内容流水线**：草稿 + 人审看板。DoD：从一个空位 query 生成一篇结构化草稿，走完审批。
- **Phase 4 · 发布 / feed + 监测 + 报表 + 调度**：合上整个闭环。DoD：定时重跑、KPI 上 Notion、排名掉了能告警。

## 7. 技术约束

- 语言 / 栈你定（建议 Python 后端 + 任务调度；前台用 Notion 即可，能省一个自建前端）。
- API-first；secrets 走环境变量 / 密钥管理，绝不进仓库。
- adapter 限速 + 重试 + 缓存；抓取守 ToS。
- 每个模块可单测；证据计算函数必须有测试。

## 8. 现在就做（Claude Code 的第一步）

1. 读完本 brief，复述你对领域模型和红线的理解，让我确认无误。
2. 搭 Phase 0 脚手架 + 定死 Evidence schema。
3. 接一个 adapter，对礼盒品类跑一个最小的端到端侦察切片（中文一条 + 英文一条），把真实证据给我看——先证明数据是真的，再继续往上盖。
4. 然后停下来等我看 Phase 0 结果，再开 Phase 1。
